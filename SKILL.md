# 文档排版助手 (docx-helper)

对已有的 Word 文档 (.docx) 按规范排版习惯（参考 GB/T 9704-2012）自动调整格式。

## 核心理念

三个设计原则贯穿整个工作流：

1. **先重置、再判结构、后套用**：绝不在脏格式上叠加。先用 `--reset` 清空文档所有手动格式（含自动编号 numPr），再由大模型通读全文判定结构，最后按结构套用。这样能根治"标题被误判为正文""序号字体与正文不一致"等老问题。
2. **大模型判结构，正则只兜底**：段落类型（标题层级）由大模型依据全文上下文判定，而非逐段孤立的正则匹配——无编号的小节标题（如"编制背景"）正是正则的盲区，必须靠理解上下文。
3. **版本化输出**：每次排版输出 `原名+docx-helper+v{N}.docx`，原文件永不覆盖，新旧版本链式追溯。

## 标准工作流

```
① 重置格式          ② 大模型判结构        ③ 确认建议          ④ 套用排版
──────────        ──────────          ──────────        ──────────
--reset 原文  →    通读全文判定段落类型  →  展示结构映射    →  --apply --structure
清空所有手动格式      (title/h1~h4/body)      用户确认/调整      输出版本化文件
```

### 第一步：重置格式

把原文所有手动格式一键清空，得到一个"中性文档"——仅保留文字与段落顺序，字体/字号/颜色/加粗/缩进/对齐/自动编号全部还原：

```bash
python scripts/format.py --reset 报告.docx
# 输出：报告+reset.docx
```

可选：打印所有非空段落的「索引: 文本」，便于下一步判结构：

```bash
python scripts/format.py --list 报告+reset.docx
```

### 第二步：大模型判结构（关键）

阅读 reset 文档全文（或 `--list` 输出），**按语义深度而非编号字符**判定每个段落的类型：

| 类型 | 语义 | 字体 | 判定规则（按深度，不看具体编号字形） |
|------|------|------|------|
| `title` | 大标题 | 小标宋二号 | 文档首页的居中短句大标题 |
| `chapter` | 章 | 黑体三号 | 文档最顶层分章：`一、` `二、` … `九、`，`section`/`subsection`/`item` 的父级 |
| `section` | 节 | 楷体三号 | **章的直属子级**——无论写成 `1.` / `2.1` / `（一）`，只要是章下面的第一级子标题 |
| `subsection` | 子节 | 仿宋三号 | **节的直属子级**——如 `4.1.1`、`8.1.1` 等三层编号 |
| `item` | 条目 | 仿宋三号 | `（1）` `（2）` `①` 等带括号/圈号的编号项（通常是一段有内容的编号列表项） |
| `bullet` | 项目符号 | 仿宋三号 | `·` `●` `◆` 等符号引导的短项 |
| `body` | 正文 | 仿宋三号 | 其余——无编号的正文段落、纯文字说明 |

> **核心理念（铁律）**：不看编号写什么字符，看它在文档树里是"谁的子级"。
> `1.编制背景` 和 `2.1 盘点目标` 同为「章的直属子级」→ 均判 `section`（楷体）；
> `4.1.1 按分层迁移` 是「section 的子级」→ 判 `subsection`（仿宋）；
> `（1）…` 是带编号的内容条目 → 判 `item`（仿宋，无缩进）。

把判定结果写成 `structure.json`：

```json
{
  "paragraphs": {
    "0": "chapter",
    "1": "section",
    "3": "section",
    "5": "item",
    "12": "section",
    "16": "subsection"
  },
  "cover": false,
  "title_index": null
}
```

> **旧名兼容**：`h1`/`h2`/`h3`/`h4` 仍可接收，脚本内部自动映射。
> 即 `"h1"` 等同于 `"chapter"`、`"h2"` 等同于 `"section"`，以此类推。

### 第三步：确认建议

把结构映射摘要（各类型段落数、封面判断）展示给用户确认。用户可调整，例如：

> "第 12 段应该是 section 不是 body" —— 改 `structure.json` 后重跑即可。

### 第四步：套用排版

```bash
# 按结构映射套用（推荐）
python scripts/format.py --apply 报告+reset.docx --structure structure.json
# 输出：报告+reset+docx-helper+v1.docx

# 指定版本号
python scripts/format.py --apply 报告+reset.docx --structure structure.json --version 3
```

不加 `--structure` 时，脚本回退到旧版正则启发式（仅兜底，不推荐）。

### 自动检测模式（P2）— 一条命令搞定

对于大多数标准编号文档（GB 或 x.y 体系），一条命令即可自动排版，无需手动判结构：

```bash
# 自动检测编号层级 + 排版
python scripts/format.py --apply 报告+reset.docx --auto

# 仅生成 structure.json 初稿（供审阅/修改）
python scripts/format.py --auto 报告+reset.docx --output structure.json
```

`--auto` 按以下规则自动推断层级：

| 编号模式 | 判为 | 说明 |
|---------|------|------|
| `一、二、…` | `chapter` | |
| `1.` `2.1` `3.1` … | `section` | **统一为同级**——无论有无章号前缀 |
| `（一）` 出现时 → `1.` 降为 `subsection` | 兼容纯 GB 文档 |
| `4.1.1`（三层数字） | `subsection` | |
| `（1）①` | `item` | |
| `·●◆` | `bullet` | |
| 其余 | `body` | |

> 覆盖约 80% 的标准/混合编号文档。不规则文档（如大量无编号标题）建议仍用 LLM 判结构 + `--structure`。

### 迭代工作流

```
v1: 报告.docx → reset → 判结构 → 确认 → 报告+reset+docx-helper+v1.docx
v2: 微调 structure.json → 报告+reset+docx-helper+v2.docx
v3: ...
```

每次都基于同一份 reset 文档继续调整，链式追溯，随时可回退到任意版本。

## 自定义配置

可通过 `.docx-helper.json` 覆盖任意排版参数（不改的部分自动沿用 GB 默认）：

```bash
# CLI 指定
python scripts/format.py --apply 报告+reset.docx --structure structure.json --config .docx-helper.json

# 自动查找（按优先级）
# 1. --config 参数显式指定
# 2. 当前目录 .docx-helper.json
# 3. ~/.workbuddy/docx-helper.json
# 4. 以上都没有 → 使用默认 GB-9704 配置
```

配置示例（只改想改的，其余继承默认）：

```json
{
  "fonts": {
    "section": "思源宋体",
    "body": "思源宋体"
  },
  "sizes": {
    "body": 14
  },
  "spacing": {
    "body_indent_chars": 2
  }
}
```

完整默认配置见项目根目录 `docx-helper-default.json`（即 `.docx-helper.json`）。

## 排版规则

### ⚠️ 格式优先级（铁律）

所有排版判断按以下优先级执行，**上一级有明确规定的绝不查下一级**：

```
① 用户提供的排版手册/截图（最高优先，本文档的基石）
② GB/T 9704-2012《党政机关公文格式》（仅当 ① 没有覆盖时使用）
③ 脚本内置默认值（仅当 ①② 均无规定时兜底）
```

**① 的现行内容**（依据用户截图，全文排版以此为第一准则）：

- 页面：A4，上 3.7cm / 下 3.5cm / 左右 2.7cm，页脚距底边 2.54cm
- 标题：2 号方正小标宋_GBK（大标题上方空 2 行、下方空 1 行）
- 正文行距：28.9 磅（固定值）
- 字体体系：一级标题 3 号方正黑体_GBK，二级标题 3 号方正楷体_GBK，正文 3 号方正仿宋_GBK
- 西文/数字：一律 3 号 Times New Roman
- 页码：奇数页右下角、偶数页左下角，4 号 Times New Roman，格式 `— N —`

**② 的使用范围**：① 中没有规定的（如封面页排版详情、多级编号的（一）/1./（1）层级字体），才去查 GB/T 9704-2012。① 和 GB 冲突时，以 ① 为准。

> 示例：GB 允许页码居中，但 ① 规定奇数右下偶数左下 → 用 ①。
> ① 和 GB 都没说的地方（如超四层编号的字体），脚本用内置默认值。

### 页面

| 项目 | 值 |
|------|-----|
| 纸张 | A4 (210 × 297 mm) |
| 上边距 | 3.7 cm |
| 下边距 | 3.5 cm |
| 左/右边距 | 2.7 cm |
| 页脚距底边 | 2.54 cm |

### 字体（方正字库） — 结构类型 → 字体映射

| 结构类型 | 语义 | 字体 | 字号 | 加粗 |
|---------|------|------|------|------|
| `title` | 大标题 | 方正小标宋_GBK | 二号 (22pt) | 否 |
| `chapter` | 章标题 | 方正黑体_GBK | 三号 (16pt) | 否 |
| `section` | 节标题 | 方正楷体_GBK | 三号 (16pt) | 否 |
| `subsection` | 子节标题 | 方正仿宋_GBK | 三号 (16pt) | 否 |
| `item` | 条目编号 | 方正仿宋_GBK | 三号 (16pt) | 否 |
| `body` | 正文 | 方正仿宋_GBK | 三号 (16pt) | 否 |
| 西文/数字 | — | Times New Roman | 三号 (16pt) | — |
| 页码 | — | Times New Roman | 四号 (14pt) | — |

> 这是**默认映射**（参考 GB/T 9704-2012）。用户可通过 `.docx-helper.yml` 自定义覆盖。
> **旧名兼容**：`h1`→`chapter`、`h2`→`section`、`h3`→`subsection`、`h4`→`item`。

### 段落

| 项目 | 值 |
|------|-----|
| 正文行距 | 28.9 磅（固定值） |
| 正文首行缩进 | 2 字符 |
| 大标题上方 | 空 2 行 |
| 大标题下方 | 空 1 行后开始正文 |

### 标题识别（语义深度判定，不按编号字形匹配）

判定规则见上方「大模型判结构」步骤。**不看编号写什么字符，看它是"谁的子级"。**

GB/T 9704-2012 §7.3.3 的四层序数体系（`一、`→`（一）`→`1.`→`（1）`）是一种**常见的编号惯例**，但不是唯一写法。实际文档中的 `1.编制背景` 和 `2.1 盘点目标` 应判为同类（`section` 节标题，楷体），`4.1.1` 判为 `subsection`（子节标题，仿宋）。

| 层级 | 编号格式 | 字体 | 合规性 |
|------|---------|------|--------|
| 一级 | `一、二、三、` | 黑体 | ✅ 国标第一层 |
| 二级 | `（一）（二）` | 楷体 | ✅ 国标第二层 |
| 三级 | `1. 2. 3.` | 仿宋 | ✅ 国标第三层 |
| 四级 | `（1）（2）` `①②` | 仿宋 | ✅ 国标第四层 |
| — | `● ◆ ★ ■` | — | ❌ 不在国标内，正式公文不应使用 |
| — | `1.1 4.1.1` | — | ❌ 不在国标内，技术方案可用 |

分析报告会逐段列出每处不合规编号，并给出修改建议（如：`●` 建议改为 `（1）` 格式）。

### 封面排版（材料/方案/报告）

GB/T 9704-2012 不涉及封面，以下为各地政府机关实际使用的惯例：

| 元素 | 字体 | 字号 | 位置 | 间距 |
|------|------|------|------|------|
| 大标题 | 方正小标宋_GBK | 二号 | 居中 | 上方空 7 行（约页面 1/3 处） |
| 副标题 | 方正楷体_GBK | 三号 | 居中，主标题正下方 | 与主标题同行距，不空行，可前加"——" |
| 单位名称 | 方正仿宋_GBK | 三号 | 居中 | 与标题区之间空 5 行 |
| 日期 | 方正仿宋_GBK | 三号 | 居中，单位正下方 | 同行距，不空行 |

封面排版后会在此页末尾自动添加分页符，正文从第二页开始。如果文档不需要封面排版，确认阶段告知即可跳过。

### 页码

- 格式："— N —"（em dash + 页码 + em dash）
- 脚本默认使用**居中**页码（python-docx 对奇偶页不同页脚支持有限）
- 如需精确实现"奇数页右下、偶数页左下"，排版后加载 `docx` 技能，通过 XML 添加 `w:evenAndOddHeaders` + `w:footerReference w:type="even"` 来实现

### 表格

GB/T 9704-2012 对表格内容格式无强制细化规范。脚本使用正文格式（方正仿宋_GBK 三号、无首行缩进）作为默认处理。

| 区域 | 常见做法 | 字号 |
|------|---------|------|
| 表头 | 黑体 | 小四 (12pt) 或 五号 (10.5pt) |
| 表体 | 仿宋/宋体 | 五号 (10.5pt) |

> 如果文档中对表格字号有特殊要求（如"表头黑体小四、表体仿宋五号"），请在确认阶段告知，用覆盖规则或 docx 技能手动调整。

## 精细调整（脚本覆盖不到的场景）

脚本覆盖了页面、字体、行距、页码、表格等常规排版。以下场景需要用系统内置 `docx` 技能手动处理：

- **奇偶页页码位置**：脚本默认居中；需改为奇数页右下、偶数页左下
- 版头元素：红色分隔线、发文机关标志、发文字号、签发人
- 印章位置：发文机关署名 + 成文日期 + 印章布局
- 版记：抄送机关、印发机关和日期、分隔线
- 多行标题的梯形或菱形排列
- 附件的编排格式、页码连续性

加载 `docx` 技能后，用其 unpack → 编辑 XML → repack 工作流进行精细调整。详细规则见 `references/gbt9704_2012.md`。

## 补充规则（来自 GB/T 9704-2012）

当用户材料要求未覆盖时使用。完整内容见 `references/gbt9704_2012.md`。

- 主送机关：标题下空一行，居左顶格，末尾全角冒号
- 附件说明：正文下空一行左空二字"附件："，名称后不加标点
- 成文日期：右空四字，阿拉伯数字标全年月日，不编虚位
- 印章：红色，居中下压署名和日期
- 装订：左侧装订，两钉距上下边缘各 70 mm

## 注意事项

- **仅支持 .docx**：旧版 .doc 需先另存为 .docx
- **字体前置条件**：系统需安装方正字库（方正小标宋/黑体/楷体/仿宋_GBK），否则 Word 会提示字体缺失
- **版本安全**：原文件永不修改，每次输出新版本文件
- **表格处理**：脚本用正文字体但取消首行缩进；如需保留表格特殊格式，排版后手动调整
